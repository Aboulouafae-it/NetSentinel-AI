Wireless Link Intelligence

Executive Decision & Implementation Strategy Document

For Engineering Leadership, Architecture, and Product Management

1. Executive Decision Summary

The Strategic Mandate: Wireless Link Intelligence (WLI) must be built as a highly normalized, hardware-agnostic subsystem within NetSentinel AI. It will translate analog RF physics into deterministic digital states.

The Execution Posture: Implementation will aggressively favor data integrity and manual oversight over premature automation. We will build a rigid data schema first, simple deterministic alerting second, and AI-assisted likely-cause analysis third. Automated remediation is explicitly deferred to later phases to protect operator trust.

3. MVP Inclusion vs. Exclusion Boundaries

Include in MVP

  • PTP and PTMP link modeling logic.
  • Ingestion of core 4 metrics: RSSI, SNR, CCQ, Tx/Rx Capacity.
  • Calculation of "Expected RSSI" vs "Actual RSSI".
  • Deterministic alerting (e.g., "SNR dropped > 10dB").
  • Manual field visit logging (text notes & status).
  • Read-only integration with global Incident system.

Exclude from MVP (Defer)

  • Automated topology discovery (LLDP/CDP parsing).
  • Mesh Wi-Fi roaming analytics.
  • Spectrum scan ingestion and visualization.
  • AI Copilot generating physical configuration commands.
  • Automated creation of links from Edge Agent scans.
  • Mobile App for field technicians.

5. Recommended Implementation Sequence

  1. Phase 1: The Schema Foundation. Implement the core database entities (Sites, Assets -> Radio Interfaces -> Wireless Links). Define the TSDB hyper-tables for RF metrics.
  2. Phase 2: The Data Pipe. Build the generic SNMP poller in the Edge Agent. Build the normalization worker that translates vendor OIDs into the unified TSDB schema.
  3. Phase 3: The Deterministic Engine. Implement the rule-based alert matrices (Hold-down timers, topological suppression, Expected vs Actual RSSI deltas).
  4. Phase 4: The Intelligence Layer. Feed the deterministic outputs into the AI Copilot to generate plain-English likely-cause summaries and pre-visit field briefs.

6. Core Entities That Must Exist First

Do not attempt to ingest data until these primitives are stable in the database:

Entity Primitive Core Attributes Architectural Purpose
PhysicalMount Elevation, Azimuth, Tower ID Differentiates devices on the same tower to isolate wind-sway vs global power failure.
RadioInterface MAC, Frequency, TX Power, Channel Width Decouples the hardware from the link. Allows for radio swapping without losing historical link data.
AntennaProfile Gain (dBi), Beamwidth, Polarization Essential math variable required to calculate theoretical Expected RSSI.
WirelessLink Interface A, Interface B, Link Type (PTP/PTMP) The core logical relationship that the TSDB and AI Copilot bind telemetry to.

7. Telemetry Sources That Can Wait

Prioritize raw survival metrics. Advanced diagnostic data should be deferred to avoid ingestion bottlenecking.

8. Operational Reality: What Stays Manual Early

Automation requires absolute trust. We will rely on manual workflows to build that trust.

  • Link Topology Creation: Users will manually map Site A to Site B. Automated LLDP discovery is too brittle for MVP.
  • Expected RSSI Baselines: Users will input theoretical targets based on their own link planners. System will not calculate complex free-space path loss (FSPL) internally yet.
  • Field Visit Validation: Technicians manually click "Validate Repair." The system will not automatically close tickets just because RSSI improved.

9. Anti-Patterns: What Not to Attempt Early

  • Automated Remediation: Never allow the system to automatically change frequencies, reboot radios, or alter channel widths in early phases. A mistake here drops a network offline and destroys operator trust permanently.
  • Direct AI Metric Analysis: Do not feed raw float streams directly to an LLM. It is slow, expensive, and prone to hallucination. Always use deterministic logic gates first.
  • Vendor-Specific DB Columns: Never add columns like ubiquiti_quality to the core schema. Normalize all vendor data into standard RF metrics in the ingestion pipeline.

10. Final Architectural Guidance

Keep it Powerful Without Over-Engineering:

The success of the Wireless Link Intelligence pillar depends entirely on Data Normalization and Stateful Alerting. If the system treats a MikroTik and a TP-Link identically in the database, the UI and AI layers become trivial to build.

Do not build complex machine learning models for anomaly detection yet. A simple threshold alert combined with a hold-down timer (e.g., "SNR < 20dB for > 5 minutes") covers 90% of operational use cases. Focus engineering effort on ensuring the Edge Agent never drops polling data, and the TimescaleDB handles the write-load efficiently. The intelligence of the system emerges from reliable data, not complex math.

التفويض الاستراتيجي: يجب بناء "ذكاء الروابط اللاسلكية" (WLI) كنظام فرعي مستقل عن الأجهزة وموحد البيانات داخل NetSentinel AI. سيقوم بترجمة الفيزياء التناظرية للترددات الراديوية (RF) إلى حالات رقمية حتمية.

موقف التنفيذ: سيفضل التنفيذ بقوة سلامة البيانات والإشراف اليدوي على الأتمتة المبكرة. سنقوم ببناء مخطط بيانات صارم أولاً، وتنبيهات حتمية بسيطة ثانياً، وتحليل الأسباب المحتملة بمساعدة الذكاء الاصطناعي ثالثاً. تم تأجيل المعالجة الآلية صراحةً للمراحل اللاحقة لحماية ثقة المشغلين.

3. حدود نطاق الحد الأدنى للمنتج (MVP)

التضمين في المرحلة الأولى (MVP)

  • منطق نمذجة الروابط PTP و PTMP.
  • استيعاب المقاييس الأساسية: RSSI، SNR، CCQ، سعة Tx/Rx.
  • حساب "الـ RSSI المتوقع" مقابل "الـ RSSI الفعلي".
  • التنبيهات الحتمية (مثل: "انخفض SNR بأكثر من 10dB").
  • تسجيل الزيارات الميدانية يدوياً (ملاحظات نصية).
  • تكامل للقراءة فقط مع نظام الحوادث العالمي.

الاستبعاد (مؤجل للمستقبل)

  • الاكتشاف الآلي لطبولوجيا الشبكة (LLDP/CDP).
  • تحليلات التجوال في شبكات Wi-Fi المتداخلة (Mesh).
  • استيعاب وعرض مسح الطيف الترددي مباشرة.
  • توليد مساعد الذكاء الاصطناعي لأوامر تكوين مادية.
  • الإنشاء الآلي للروابط عبر مسوحات وكيل الحافة.
  • تطبيق هاتف محمول للفنيين الميدانيين.

5. التسلسل الموصى به للتنفيذ

  1. المرحلة 1: التأسيس. تنفيذ كيانات قاعدة البيانات الأساسية (المواقع، الأصول -> الواجهات الراديوية -> الروابط اللاسلكية). تحديد جداول TSDB.
  2. المرحلة 2: مسار البيانات. بناء وكيل استطلاع SNMP. بناء عامل التوحيد الذي يترجم بيانات الموردين إلى مخطط TSDB الموحد.
  3. المرحلة 3: المحرك الحتمي. تنفيذ مصفوفات التنبيه المبنية على القواعد (مؤقتات التثبيت، كبح التنبيهات الطوبولوجية، حسابات RSSI الفعلي والمتوقع).
  4. المرحلة 4: طبقة الذكاء. تغذية المخرجات الحتمية لمساعد الذكاء الاصطناعي لتوليد ملخصات الأسباب المحتملة بلغة بسيطة للفنيين.

6. الكيانات الأساسية التي يجب توفرها أولاً

لا تحاول استيعاب البيانات حتى تستقر هذه البديهيات في قاعدة البيانات:

الكيان الأساسي السمات الأساسية الهدف المعماري
حامل التثبيت PhysicalMount الارتفاع، الزاوية، معرف البرج يفصل بين الأجهزة على نفس البرج لعزل أعطال تمايل الرياح مقابل انقطاع الطاقة العالمي.
الواجهة الراديوية RadioInterface MAC، التردد، قوة البث، عرض القناة يفصل الأجهزة المادية عن الرابط، مما يسمح بتبديل الراديو دون فقدان البيانات التاريخية.
ملف الهوائي AntennaProfile الكسب (dBi)، عرض الشعاع، الاستقطاب متغير رياضي أساسي مطلوب لحساب الـ RSSI المتوقع النظري.
الرابط اللاسلكي WirelessLink الواجهة أ، الواجهة ب، نوع الرابط العلاقة المنطقية الأساسية التي ترتبط بها بيانات القياس ومساعد الذكاء الاصطناعي.

7. مصادر القياس التي يمكن تأجيلها

أعط الأولوية لمقاييس البقاء الأساسية. يجب تأجيل بيانات التشخيص المتقدمة لتجنب اختناقات الاستيعاب.

يُستوعب فوراً (MVP) يُؤجل للمراحل اللاحقة
قوة الإشارة (RSSI) مسوحات الطيف المباشرة (FFT Arrays)
مستوى الضوضاء (Noise Floor) نسب عدالة وقت البث لكل عميل
سعة البث/الاستقبال (MCS Index) أسباب سقوط الحزم العميقة (فيضان الذاكرة مقابل الترددات)

8. الواقع التشغيلي: ما يبقى يدوياً في البداية

تتطلب الأتمتة ثقة مطلقة. سنعتمد على مهام سير العمل اليدوية لبناء هذه الثقة.

  • إنشاء طوبولوجيا الروابط: سيقوم المستخدمون بتعيين الموقع أ إلى الموقع ب يدوياً.
  • خطوط الأساس للـ RSSI: سيقوم المستخدمون بإدخال الأهداف النظرية بناءً على أدوات التخطيط الخاصة بهم.
  • التحقق من الزيارة الميدانية: ينقر الفنيون يدوياً على "التحقق من الإصلاح". لن يغلق النظام التذاكر آلياً لمجرد تحسن الـ RSSI.

9. أنماط يجب تجنبها: ما لا يجب محاولته مبكراً

  • المعالجة التلقائية: لا تسمح أبداً للنظام بتغيير الترددات أو إعادة تشغيل أجهزة الراديو آلياً في المراحل المبكرة. خطأ واحد هنا سيؤدي إلى إسقاط الشبكة وتدمير الثقة.
  • التحليل المباشر للذكاء الاصطناعي: لا تقم بتغذية تيارات القيم الرقمية الخام مباشرة إلى نموذج لغوي (LLM). استخدم البوابات المنطقية الحتمية دائماً أولاً لتجنب "هلوسة" الذكاء الاصطناعي.
  • أعمدة بيانات مخصصة للموردين: لا تضف أبداً أعمدة مثل ubiquiti_quality إلى المخطط الأساسي. قم بتوحيد جميع البيانات إلى مقاييس قياسية في مسار الاستيعاب.

10. التوجيه المعماري النهائي

حافظ على القوة دون تعقيد هندسي مفرط:

يعتمد نجاح ركيزة "ذكاء الروابط اللاسلكية" كلياً على توحيد البيانات و التنبيهات ذات الحالة. إذا تعامل النظام مع جهاز MikroTik وجهاز TP-Link بشكل متطابق في قاعدة البيانات، فإن بناء واجهات المستخدم وطبقات الذكاء الاصطناعي يصبح أمراً سهلاً وموثوقاً.

لا تبني نماذج تعلم آلي معقدة لاكتشاف الشذوذ بعد. إن التنبيه البسيط القائم على العتبة مع مؤقت التثبيت (مثل: "SNR < 20dB لأكثر من 5 دقائق") يغطي 90٪ من حالات الاستخدام التشغيلي. ركز الجهد الهندسي على ضمان عدم فقدان البيانات، وتعامل قاعدة TimescaleDB مع أحمال الكتابة بكفاءة. ذكاء النظام ينبع من البيانات الموثوقة، وليس الرياضيات المعقدة.
