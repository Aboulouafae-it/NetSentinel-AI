1. Executive Summary & Product Vision

NetSentinel AI is a next-generation network management, observability, and cybersecurity platform. It is designed to bridge the gap between standard IT infrastructure monitoring and specialized radio/wireless diagnostics.

By leveraging advanced Artificial Intelligence, it empowers network operators to rapidly diagnose complex incidents—whether they originate from a sophisticated cyber threat, an internal misconfiguration, or an environmental factor affecting outdoor wireless links.

2. Architectural Overview

The platform follows a modular, API-first architecture designed for long-term scalability. It currently utilizes a monorepo approach with a clear separation of concerns across the presentation, business logic, and data layers.

3. Core Platform Pillars

🛡️ Core Asset Inventory

Comprehensive discovery, management, and tracking of routers, switches, servers, endpoints, and physical sites.

📊 Monitoring & Observability

Real-time uptime tracking, performance metrics, and centralized log ingestion across all registered infrastructure.

🚨 Security Detection

Advanced alerting, anomaly detection rules, Indicators of Compromise (IoC) tracking, and structured incident response workflows.

🧠 AI Copilot

Generative AI integration for automated incident summarization, likely-cause analysis, and chat-based troubleshooting assistance.

📡 Wireless Intelligence

Deep diagnostics for Point-to-Point (PTP) and Point-to-Multipoint (PTMP) links. Tracks RSSI, SNR, noise floor, and alignment states.

⚡ Automation Engine

Playbook rules executing automated response actions such as webhooks, asset isolation, or incident creation based on detected severities.

4. Data Model & Entity Relationships

7. Project Environment, Files & Access Scope

Environment Details

The platform runs in a modern, containerized environment defined via Docker Compose. The environment is partitioned into several independent but interconnected services:

  • Database: PostgreSQL 16 instance accessible internally by the backend.
  • Cache/Queue: Redis 7 instance for fast caching and async task processing.
  • Backend API: Python 3.12 FastAPI container running on port 8000.
  • Frontend UI: Next.js 14 container running on port 3000.
  • Background Worker: ARQ worker instance for background processing.
  • Edge Agent: A scalable local poller for internal network telemetry gathering.

Access Scope & Security Boundaries

The platform is designed with explicit boundaries regarding what data and systems it can access:

  • Internal Network Scope: Through the Edge Agent, the system can discover and poll assets within user-defined IP subnets via ICMP, SNMP, and vendor-specific APIs.
  • External Intelligence APIs: The system is equipped to reach out to external threat intelligence feeds and the generative AI APIs (e.g., Gemini) for Copilot features.
  • Data Isolation: All data is strictly segregated per Organization ID at the database layer, ensuring cross-tenant privacy.
  • Credential Management: The system accesses databases and external systems exclusively via securely stored environment variables; no plaintext credentials are committed to the repository files.

8. Legislator's Conclusions & Strategic Next Steps

System Evaluation: The NetSentinel AI platform demonstrates a robust, enterprise-grade foundation. The architectural choices—specifically utilizing FastAPI for high-throughput asynchronous telemetry ingestion, and Next.js for a responsive, SEO-friendly, and highly secure frontend interface—are perfectly aligned with modern industry standards.

Security Posture: The system implements a "secure-by-design" methodology. Authentication relies on stateless JWTs with strict expiration. The database schema strictly enforces multi-tenancy at the ORM layer, preventing cross-organization data leakage. Furthermore, the newly introduced Threat Detection models (IoCs and Detection Rules) establish the groundwork for real-time cyber defense.

Strategic Recommendations & Next Steps:

  1. Phase 8 Achieved: The implementation of a standalone Desktop Client via Electron entirely decouples the platform from browser dependence, while the custom "White Hat Snake" branding cements a highly professional, enterprise-grade identity.
  2. Fix ARQ Worker Dependencies: The UI, Desktop App, and Backend API are fully functional and polished. However, the background ARQ worker requires binary dependencies (`arq` executable) which need to be properly added to the `backend/Dockerfile` to allow background job routing to function natively in the Docker cluster.
  3. Agent Deployment & Scale Testing: Prioritize the deployment of the Edge Agent (Poller) to capture live telemetry from distributed subnets in a staging environment. We need to stress-test the TimescaleDB insertion rates for high-frequency RF polling.
  4. Mobile/Field App: Begin planning a lightweight progressive web app (PWA) or mobile app specifically for field technicians to view the AI Pre-visit Briefs generated by the Copilot while on the tower.
